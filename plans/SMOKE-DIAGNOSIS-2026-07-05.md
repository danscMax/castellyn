# Диагностика живого смоука 2026-07-05 — 11 находок владельца

Каждый вердикт проверен по коду/конфигам лично (file:line). Вероятности — оценка причин.
Root-cause фиксы без костылей, с трейдофами.

---

## A. serena: «blocked by your organization's policy» ×9 профилей → «Готово (код 0)»

**Причины:**
- **~95% — это наш же Ф2-карантин, работающий по назначению.** `config/managed-settings.json:87`
  = `"serena@claude-plugins-official": false` (задеплоено в %ProgramFiles% с UAC). CC трактует
  managed=false как организационную блокировку → user-scope `plugin enable` отбивается. Castellyn
  честно попытался включить в 9 профилях то, что сам же заблокировал уровнем выше.
- ~5% — прочее (битый managed-деплой); признаков нет.

**Root fix (2 части):**
1. Плагины-таб должен ЗНАТЬ managed-состояние: у строки с managed=false — бейдж
   «заблокирован политикой» и вместо 9 бесполезных enable — одно действие «Разблокировать
   в источнике» (правка `config/managed-settings.json` + существующий `run_managed_deploy` UAC).
   Трейдоф: UAC-запрос на каждое разблокирование; +одно чтение managed-источника в read-путь.
2. **Ложный «Готово (код 0)»**: обёртка enable глотает per-profile ошибки CLI → exit 0 → зелёный
   тост. Скрипт/нативный путь должен считать failed и отдавать status=error в envelope.
   Трейдоф: правка внешнего PS (Manage-Plugin) — вне git-дерева, покрыто Syncthing/бэкапом.

## B. max@max-marketplace: «not installed at scope user»

**Причины:**
- **~90% — scope-mismatch.** `claude plugin update` по умолчанию scope=user (проверено
  `--help`: `-s, --scope user|project|local|managed, default: user`), а max включён на scope
  **managed** (лог апдейтера: «plugins: 26 (19 managed, 7 user)»). Update юзер-скоупа для
  managed-плагина закономерно падает.
- ~10% — рассинхрон installed_plugins.json (нет записи user-scope при живом кэше).

**Root fix:** scope-aware вызовы: определить scope плагина (installed_plugins.json /
`claude plugin list`) и звать `claude plugin update -s <scope>`. Точки: наш `run_plugin`
(update/enable/disable), `Check-MarketplaceVersions.ps1 -Bump` (авто-update после бампа),
скрипт update-plugins (массовое обновление — иначе 19 managed-плагинов «обновляются» вхолостую!).
Плюс тот же envelope-фикс ложного успеха, что в A2.
Трейдоф: managed-scope операции могут требовать elevation — проверить один раз вживую; если да,
шлюзовать через UAC-путь как run_managed_deploy.

## C. Раскрытие строки плагина «бесполезно» (простыня чипов)

Не баг — UX-долг. Root fix: заменить чипы на структурированную панель деталей:
версия/скоуп/маркетплейс/путь + **включённость по профилям** (данные уже есть у матрицы V2 —
переиспользовать) + действия (bump ⇪ / update / открыть папку) + компоненты (скиллы/команды/
агенты) сворачиваемыми группами с количеством, а не стеной. Трейдоф: заметная UI-работа
(~компонент деталей), дизайн-гейт на макет.

## D. Уведомления открываются сверху, а не у колокольчика

Подтверждено: `NotificationPanel.svelte:75` — `position: fixed; inset: 0`, панель прибита к
верхнему краю, а колокольчик — внизу сайдбара. Root fix: якорить панель к кнопке через
существующий примитив `use:anchored` (`src/lib/floating.ts`). Трейдоф: нет, чистая правка.

## E. Нет «Назад» (и боковая кнопка мыши не работает)

SPA без истории переходов. Root fix: стек посещённых табов в `+page.svelte` (переключение и так
центральное) + `pointerup` с `button===3/4` (Back/Forward мыши) + Alt+←/→; опционально стрелка ←
в шапке. Трейдоф: не перехватывать события внутри xterm-терминалов и полей ввода.

## F. Главная «Задачи: 3 выключено» ↔ в Расписании непонятно, что именно

Подтверждено: `HomeTab.svelte:129` считает `!x.enabled`, куда попадают и «не создано»
(3 карточки «не создано» = те самые «3 выключено»). Root fix: (а) на Главной разделить
«{n} выключено» и «{m} не создано»; (б) в Расписании амбер-подсветка карточек, которые дали
бейдж на Главной (сейчас «не создано» — нейтрально-серое, глаз не цепляет). Трейдоф: только UI.

## G. Syncthing «не найден/не запущен», но файлы «в синхроне»

Два независимых индикатора: статус процесса/API Syncthing (`sync.ts:46` «Syncthing не найден
или не запущен») и файловое сравнение конфигов («в синхроне» — прямое сравнение содержимого,
без Syncthing). Вероятные причины расхождения:
- ~60% — детекция ищет не там, где Syncthing реально запущен (сервис vs tray-процесс vs
  SyncTrayzor), при живом бинаре.
- ~30% — REST API не отвечает на ожидаемом адресе/ключе (родственник Дыры4).
- ~10% — таймаут/права.
Root fix: один источник истины — ping REST API по адресу из живого конфига Syncthinga, процессный
чек только как fallback; и согласовать формулировки (Главная показывает «idle» — это статус API,
таб говорит «не запущен» — это другой чек). Нужен 15-минутный рекон текущего детекта + твой факт:
как именно запущен Syncthing на этой машине. Трейдоф: нет.

## H. Codex: не подтянулись профили и нет суффиксов типа --yolo

По построению: профиль применяется только к claude (`lib.rs:11594` — «profile only matters for
claude», CLAUDE_CONFIG_DIR), чипы аргументов — общие, заточены под claude
(`--dangerously-skip-permissions` codex'у бесполезен). Root fix двумя слоями:
1. **Per-env пресеты аргументов** (дёшево): у codex свои чипы (`--yolo`, `--full-auto`,
   `--profile <name>`…), у opencode свои; дефолт-аргументы тоже per-env.
2. **Профили codex** (средне): парсить `[profiles]` из `~/.codex/config.toml` (toml_edit уже
   в зависимостях, мы туда уже пишем gateway-провайдер) → дропдаун профиля для env=codex →
   `--profile <name>` в args.
Трейдоф: слой 2 добавляет чтение toml в launcher-путь; UI сегмент профиля станет env-зависимым.

## I. Codex-панель пишет «Claude profiles loaded»

Это НЕ codex с клод-профилем. Панели запускают инструмент внутри pwsh (`lib.rs:11612`: tools —
.cmd-шимы → «launch inside pwsh; -NoExit»), pwsh грузит твой PowerShell-профиль, а баннер
печатает твой же модуль claude-profiles из profile.ps1. Варианты:
- **Рекомендую:** гейтить баннер в самом profile.ps1: `if (-not $env:CASTELLYN_SESSION_ID)` —
  переменная уже инжектится в каждую панель Castellyn. Чисто, ничего не ломает.
  Трейдоф: правка внешнего дерева (не под git).
- Альтернатива `-NoProfile` для не-shell панелей — опасно: PATH-твики профиля могут быть нужны
  для резолва claude/codex шимов. Не рекомендую.

## J. Правый клик — нативное меню WebView2 («Обновить/Печать»)

Дефолтное контекстное меню webview никто не глушил. Root fix: глобальный
`contextmenu → preventDefault` с белым списком (input/textarea/contenteditable/xterm — там
нативные копипаст-меню полезны) + собственные контекст-меню там, где ждёт пользователь
(рельс агентов: фокус/переименовать/в фон/закрыть; таб проекта: rename/close). Трейдоф:
в местах без нашего меню пропадёт и нативное — покрыть главные поверхности сразу.

## K. «＋» плодит только codex; выбранный раньше клод-рецепт не предлагается

Так спроектирован лаунчер C: главный клик = рецепт ПЕРВОЙ панели активного проекта (codex был
первым → всё клонится codex), а прошлые запуски живут в меню за шевроном ▾. Но есть два реальных
дефекта дизайна:
1. **Недавние пишутся не всеми путями запуска** — только форма/избранное/недавние. spacePlus
   («＋» на табе и главный сплит), клон Ctrl+Shift+D, «Запустить стек», запуск из Профилей —
   НЕ записывают → твой клод-запуск в меню и не попал. Root fix: recordRecent на уровне
   `addPane` (единая точка всех спавнов).
2. **Рецепт проекта = первая панель, а не последний явный выбор.** Root fix: per-space
   «текущий рецепт» = последний ЯВНО выбранный запуск в этом space (форма/меню), клоны его
   не перетирают; в меню пункт «Сменить рецепт проекта…». Подпись сплита остаётся честной.
Трейдоф: чуть больше состояния (per-space рецепт в persist), зато поведение предсказуемое.

## Сквозное

- **«Готово (код 0)» при провалах** (A, B) — системная дыра доверия к exit-кодам обёрток
  плагинов; чинится в envelope-слое скриптов + парсинге факапов, Castellyn уже умеет показывать
  error-outcome.
- **Смоук шёл на exe ДО сегодняшних фиксов**: аудит-волна `81d0b53` (12 фиксов, включая
  data-loss guard settings.json) в собранный exe не попала — пересобрать перед продолжением.
- ⚠ **claude-mem worker упал** во время «Обновить всё» (в логе апдейтера видно бэкап его БД) —
  хук памяти отваливается десятки раз подряд в живых сессиях. Перезапуск воркера/новая сессия
  должны поднять; если нет — отдельный разбор.

## Предлагаемые волны

1. **«Правда о плагинах»** (A+B+сквозное): scope-aware CLI, managed-бейдж+разблокировка,
   честные exit-коды. Самая болючая по доверию зона.
2. **UI quick-wins** (D якорь уведомлений, F задачи, I баннер, G рекон синка) — мелкие, быстрые.
3. **Sessions-логика** (K недавние из addPane + per-space рецепт, H codex-пресеты, J контекст-меню,
   E назад) — средняя, дизайн-зависимая.
4. **Плагин-детали C** — отдельный UI-трек с макетом.
