# Диагностика «настоящих дыр» синхронизации + root-cause fix'ы

Дата: 2026-07-03. Всё выверено по коду (три sync-агента + личная перепроверка ключевого).
Файл-карта: `E:\Scripts\Castellyn\plans\dev-improve-0703\2026-07-03\sync-map.html`.

## Рамки и жёсткие ограничения (действуют для КАЖДОГО фикса)
1. **Должно работать И в Castellyn, И без него.** Механизм не должен требовать запущенного GUI.
   Правило: истина — в ФАЙЛАХ на диске (канон/синк-файлы), а Castellyn и любой скрипт — лишь читатели/писатели.
2. **Не ломать существующие junction'ы/симлинки, настроенные через ClaudeProfiles.**
   Правило: фиксы трогают ТОЛЬКО новые файлы или конфиги-приёмники (`opencode.json`, `config.toml`,
   `.claude.json`, `settings.json`-записи с нашим маркером). Никогда — залинкованные папки
   (`agents/commands/hooks/plugins/skills/projects`) и не пересоздавать линки.
3. **Root cause, без костылей.** Не «замазать симптом», а убрать причину; где причина — «нет
   durable-хранилища» → дать хранилище, а не автосейв-в-localStorage-почаще.

## ДЫРА 1 — Персонализация Сессий нигде не сохраняется (пункт 18) 🔴 главная

**Симптом:** сохранённые наборы сессий, избранные команды, раскладка колонок, корень проектов,
дефолтные аргументы, monitor-layout — теряются при переустановке ОС / переезде / чистке хранилища
приложения. Не в бэкапе, не в Syncthing.

**Причины и вероятность (по коду, не догадки):**
| Причина | Вероятность | Доказательство |
|---|---|---|
| Все `cmh-*` пишутся ТОЛЬКО в webview-localStorage, единственный писатель | **100% (подтверждено)** | `SessionsTab.svelte` — 14 ключей, инвентарь в `sync-C-sessions.md §2`; ни одного mirror в HubConfig/файл |
| localStorage не под `~/.claude` → Syncthing-whitelist его не видит | **100%** | whitelist = history/projects/skills/agents/commands/keybindings (`lib.rs:1792`) |
| localStorage не в снимке бэкапа (бэкап = канон + `creds/`, не webview-store) | **100%** | структура `Backups/…/main/`+`creds/`; `run_backup` `lib.rs:1332` |
| Исключение: SSH-хосты приходят с бэкенда (`readSshHosts`), они НЕ страдают | — | `SessionsTab.svelte:154` |

**Root-cause fix — варианты:**

- **A. Sidecar-файл под `~/.claude` (напр. `~/.claude/castellyn/sessions.json`) — РЕКОМЕНДУЮ.**
  Единственный вариант, закрывающий ОБА ограничения сразу:
  - *Standalone:* обычный JSON-файл, существует и читается без Castellyn.
  - *Cross-machine:* добавить путь в Syncthing-whitelist (`sync_item_lines`, `lib.rs:1792`) → едет на 2-й ПК сам.
  - *Бэкап:* попадает в снимок, т.к. лежит в бэкапимой зоне.
  - *Junction'ы целы:* это НОВЫЙ файл в отдельной подпапке, не трогает залинкованные `skills/projects/...`.
  - Модель: **файл — истина, localStorage — кэш**. На первом чтении, если файл пуст, а localStorage
    не пуст → сгенерировать файл из localStorage (миграция). Дальше пишем в файл, localStorage — зеркало.
  - **Трейд-оффы:** (1) кладём свой файл в чужую (Claude) папку — но это ровно та зона, что синкается;
    держать в подпапке `castellyn/`, чтобы не мусорить. (2) Две машины правят prefs → Syncthing создаст
    `*.sync-conflict-*`; для настроек приемлемо (последний-выигрывает + конфликт-файл виден). (3) нужен
    read-on-focus/watch, чтобы подхватить изменение с другой машины.

- **B. Зеркалить в HubConfig (`%APPDATA%\castellyn\config.json`).**
  - *Плюс:* сразу едет в бэкап/экспорт Castellyn; просто.
  - *Минус:* HubConfig НЕ под `~/.claude` → Syncthing НЕ повезёт (нет cross-machine, если отдельно не
    засинкать `%APPDATA%\castellyn`); читается только Castellyn'ом (не standalone в общем смысле).
  - Закрывает «в Castellyn», но НЕ «standalone/переезд между ПК» — слабее A по вашим ограничениям.

- **C. Гибрид: sidecar-файл + добавить его в Syncthing + отражать в бэкап.** Это и есть A с явным
  включением в оба канала. По сути — рекомендуемая реализация A.

**Итог по Дыре 1:** делаем **A** (= C): sessions.json под `~/.claude/castellyn/`, миграция из localStorage,
в Syncthing-whitelist и в бэкап. Root cause («нет durable-дома») убран; оба ограничения соблюдены.

## ДЫРА 2 — MCP/провайдеры: раздача вручную, нет индикатора расхождения, остаются «хвосты» 🔴

**Симптом:** правишь `.mcp.json`/`myproviders.json` — OpenCode/Codex об этом не узнают, пока не нажмёшь
Deploy на каждом; удалённый из канона сервер остаётся в `opencode.json`/`config.toml`; нет подсказки «устарело».

**Причины и вероятность:**
| Причина | Вероятность | Доказательство |
|---|---|---|
| Fan-out = одноразовый PUSH-командой, без реконсиляции | **100%** | `run_opencode_mcp` `lib.rs:6531`, `run_codex_mcp` `:6784`, `run_*_providers` |
| Нет удаления: merge только add/overwrite канон-имён, снятые — не убирает | **100%** | `sync-B-fanout.md §2 GAP`; merge сохраняет user-серверы (осознанно) |
| Нет drift-вью для OpenCode/Codex (для Claude-профилей — есть, `read_mcp`) | **100%** | карточки «Среды» показывают лишь счётчик (`lib.rs:6209-6238`) |
| Нет авто-триггера на изменение канона | **100%** | нет watcher/scheduler; всё — кнопки |

**Root-cause fix — варианты:**

- **A. Превратить Deploy в настоящую РЕКОНСИЛЯЦИЮ (add + update + remove по маркеру) — РЕКОМЕНДУЮ.**
  Помечать наши записи как castellyn-managed (namespace/поле); при деплое: канон = желаемый набор →
  добавить/обновить наши + **удалить наши, которых больше нет в каноне**; user-добавленные не трогать никогда.
  Убирает корень «хвостов». *Standalone:* вынести реконсиляцию в отдельный скрипт (`deploy-*.ps1`/rust-bin),
  который Castellyn вызывает, но который запускается и из терминала → работает без GUI. *Junction'ы:* пишет
  только в `opencode.json`/`config.toml`/`.claude.json` — не в залинкованные папки.
  **Трейд-офф:** нужно надёжно отличать «наши» от «user» в каждом конфиге (маркер/таблица); чуть больше логики.

- **B. Индикатор расхождения (drift-бейдж на «Среды»).** Хэш канона vs развёрнутого набора → «устарело,
  нажми Deploy». Дёшево, снимает «легко забыть». *Минус:* не чинит сам, только подсвечивает.

- **C. Авто-редеплой по изменению канона (file-watch).** *Минус:* требует запущенного Castellyn (ломает
  «standalone»); фоновые записи в чужие конфиги без спроса рискованны. **НЕ рекомендую.**

**Итог по Дыре 2:** **A + B**. Реконсиляция с маркером (убирает хвосты, root cause) + drift-бейдж (чтобы
не забывать). C отвергаем как костыль/риск. Раздача по сути — функция панели, но ядро выносим в скрипт →
и standalone, и в Castellyn.

## ДЫРА 3 — «Хвосты» хуков (пункт #5) 🟡 почти закрыто

**Уточнение по коду:** записи в `settings.json` при выключении УЖЕ вычищаются (`hook_cmd_unwire`
схлопывает пустые, `lib.rs:7960-7967` — перепроверено лично). Так что «пустых огрызков» НЕ остаётся.

**Что реально остаётся (причины):**
| Причина | Вероятность | Доказательство |
|---|---|---|
| unwire проходит ТОЛЬКО по текущим профилям → запись в профиле, удалённом/переименованном мимо Castellyn, осиротеет | **100%** | `plugin_sync_set/agent_status_hook_set` → `plugin_sync_profiles(&home)` `lib.rs:8035,:8174` |
| `.py`-файлы хуков не удаляются при выключении | **100%** | `ensure_*_script` только пишет; вы решили .py НЕ удалять |

**Root-cause fix (в рамках вашего решения «только по маркеру, чужие папки не трогать»):**
- При unwire дополнительно просканировать `~/.claude-*` каталоги, у которых есть РЕАЛЬНЫЙ `settings.json`,
  и снять ТОЛЬКО записи с нашим castellyn-маркером ВНУТРИ settings.json. Не открывать другие файлы, не
  трогать не-`.claude-*` папки (`.claude-mem` мимо). *Junction'ы:* правим только содержимое settings.json,
  структуру линков не трогаем. **Трейд-офф:** заглядываем в профили вне текущего списка — но действие
  строго «маркер внутри settings.json», что вы и утвердили; риск минимальный.
- `.py` не удаляем (ваше решение). Они инертны без вписанного хука.

**Итог по Дыре 3:** остаётся один edge-case (осиротевшие профили). Ценность низкая (редко удаляют/переименовывают
профили в обход Castellyn). **Решение за вами:** оставить крошечный фикс в списке или выкинуть #5 как почти-избыточный.

## ДЫРА 4 — Syncthing: адрес/порт зашиты в код 🟡

**Причина (100%):** `%LOCALAPPDATA%\Syncthing` и `127.0.0.1:8384` — константы (`lib.rs:2003,:2018`).
Нестандартная установка/порт → Castellyn покажет «не настроено», хотя Syncthing работает.

**Root-cause fix:** читать реальный адрес GUI из того же `config.xml` (`<gui><address>`), который уже
парсится ради apikey. **Трейд-офф:** почти нулевой; чисто читательская правка статус-вью, junction'ов не касается,
Syncthing и так standalone. Дёшево и правильно.

## ДЫРА 5 — Codex получает только шлюз-провайдер ⚪ by design, не баг

Codex говорит только на Responses-API (у `WireApi` нет `chat` с 2026-02), поэтому raw-провайдеры
(anthropic/chat-completions) там молча не работают — их сознательно НЕ раздают (`lib.rs:6835-6837`).
**Не чиним** (ограничение самого Codex). Единственное улучшение — подпись на карточке «почему у Codex
только шлюз», чтобы не выглядело как баг. Косметика, опционально.
