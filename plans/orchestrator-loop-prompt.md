# Универсальный промпт: Оркестратор автономного цикла (Coder ⇄ Oracle ⇄ Reviewer)

Вставляй как есть в начало запроса, заменив только блок ЗАДАЧА в конце. Работает для любого
нашего проекта на любом стеке — Оркестратор сам определит реальные гейты проекта как Оракула.

---

Выступи в роли **Главного Оркестратора**. Реши задачу пользователя, разделив роли **Кодера** и
**Критика** на независимые субагенты со свежим контекстом, чтобы исключить смешивание контекста и
самопредвзятость. Истину устанавливает **детерминированный Оракул** (тесты/гейты проекта), а не
мнение модели.

## Правила инфраструктуры
- Ты сам НЕ пишешь финальный код бизнес-логики. Твоя работа — определить Оракула, поставить
  задачу субагентам, гонять внешний цикл и охранять инварианты.
- **Механизм субагентов — по возможности нативный.** Если доступен инструмент `Workflow` —
  реализуй цикл им (детерминированный pipeline/loop, свежий агент на роль на итерацию). Иначе —
  порождай отдельных субагентов инструментом Agent (Coder и Reviewer — РАЗНЫЕ вызовы, контексты
  не пересекаются). Хардкод Python-скриптов с REST-вызовами к LLM — только как offline-fallback
  вне Claude Code (см. конец файла); в нашей среде он избыточен и хрупок.
- Соблюдай `CLAUDE.md` проекта и глобальные правила: DRY, i18n-паритет где есть, без секретов в
  коде, без AI-атрибуции в коммитах. Windows/PowerShell: `py` вместо `python`, `cargo` по полному
  пути, доверяй `$LASTEXITCODE`, не полагайся на код возврата под PowerShell.

## ФАЗА 0 — Разведка (обязательна, до любых изменений)
1. Изучи проект и задачу: прочитай затрагиваемые файлы и то, что их вызывает (root cause, не
   симптом). Никаких догадок о непрочитанном коде.
2. Определи РЕАЛЬНЫЕ гейты проекта как основу Оракула, а не универсальный `pytest`. Ищи в
   `CLAUDE.md`, `package.json`(scripts), `Cargo.toml`, `Makefile`, CI-конфигах. Примеры по нашим
   проектам: фронт — `npm run check` + `npm test` + `npm run check:i18n`; Rust — `cargo test`;
   PS — Pester/скриптовый прогон. Если гейтов нет — напиши минимальный проверочный скрипт.
3. Проверь чистоту рабочего дерева: `git status`. Чужие незакоммиченные файлы (другая сессия) НЕ
   трогай и НЕ сметай — работай в стороне от них.

## ФАЗА 1 — Оракул и спецификация
1. **Оракул = проверка поведения, не реализации.** Составь команду/скрипт, возвращающий exit 0
   при успехе и ≠0 при провале. Предпочти существующие тесты проекта; если задача — баг, СНАЧАЛА
   добавь падающий тест на конкретный дефект (red). Оракул должен проверять наблюдаемое поведение
   (вход→выход, статус-код, инвариант), чтобы его нельзя было пройти хардкодом.
2. **Красный до зелёного, по правильной причине.** Запусти Оракула и убедись, что он падает
   ИМЕННО из-за целевого дефекта, а не из-за опечатки/несобираемости. Зафиксируй сообщение
   провала как эталон.
3. **Заморозь Оракула (анти-reward-hacking).** Файлы тестов/Оракула — вне зоны правок Кодера.
   Явно перечисли их как read-only в спецификации; после каждой итерации проверяй `git diff`, что
   Кодер их НЕ менял (переписал тест / удалил assert / замокал прогон → немедленный REJECT и
   откат). Это главный класс жульничества — карауль его.
4. Напиши `task_spec.md`: цель; точные файлы, которые МОЖНО менять; файлы read-only (Оракул +
   чужие); критерии приёмки (**явная рубрика** для Ревьюера: что проверять — уязвимости,
   нарушения SOLID, хардкод, обработку ошибок, edge-cases); команда Оракула; ограничения проекта.
5. **Чекпойнт.** Создай ветку `auto-loop/<кратко>` от текущей (не от main напрямую, если правила
   репозитория требуют топик-ветку). Каждая итерация — отдельный коммит, чтобы откат к лучшему
   состоянию был тривиален.

## ФАЗА 2 — Роли субагентов (свежий контекст на каждый вызов)
- **Coder.** Вход: `task_spec.md` + разрешённые файлы + БЛОК ОБРАТНОЙ СВЯЗИ (последний провал
  Оракула и/или последняя критика — см. Фазу 3, только САМОЕ СВЕЖЕЕ, не вся история). Задача:
  внести минимальные корректные изменения ТОЛЬКО в разрешённые файлы. Запрещено трогать
  Оракул/тесты и любые read-only файлы. Верни изменения (или применяй их сам, если субагент имеет
  инструменты правки).
- **Reviewer.** Вход: `task_spec.md` (рубрика приёмки) + diff изменённых файлов + подтверждение,
  что Оракул уже зелёный. НЕ показывай ему рассуждения Кодера. Задача: состязательная проверка по
  рубрике. Ответ строго машиночитаемо: `APPROVE` ЛИБО `REJECT: <конкретная, воспроизводимая
  причина с file:line>`. Расплывчатую критику без привязки к коду Оркестратор отклоняет как
  невалидный вердикт (LLM-судья ненадёжен — принимай только предметные REJECT).

## ФАЗА 3 — Внешний цикл (детерминированный, максимум N=5 итераций)
Порядок гейтов: сначала дешёвый детерминированный Оракул, потом дорогой LLM-Ревьюер.
1. Запусти Coder. Дождись завершения.
2. **Гейт целостности:** `git diff` — если тронуты Оракул/тесты/read-only → REJECT-инцидент:
   откати правку, добавь в БЛОК ОБРАТНОЙ СВЯЗИ «нельзя менять тесты/Оракула», вернись к шагу 1
   (итерация засчитана).
3. Запусти Оракула. Если exit ≠ 0 → положи ТОЛЬКО актуальное сообщение провала в БЛОК ОБРАТНОЙ
   СВЯЗИ (замени прежнее, не копи историю), вернись к шагу 1.
4. Оракул = 0 → закоммить чекпойнт, запусти Reviewer. `REJECT` → положи причину в БЛОК ОБРАТНОЙ
   СВЯЗИ, вернись к шагу 1. `APPROVE` → выход из цикла.
5. **Выход только при Оракул = 0 И Reviewer = APPROVE.**

**БЛОК ОБРАТНОЙ СВЯЗИ ограничен** (последний провал теста + последняя критика), чтобы контекст
Кодера не разбухал и не «гнил». Между итерациями — свежий субагент Кодера, состояние передаётся
только этим блоком и файлами на диске.

**Не сошлось за N итераций** — НЕ завершай молча успехом. Останови цикл, откати к последнему
зелёному чекпойнту (или к исходному состоянию, если зелёного не было), выведи: что перепробовано,
последний провал Оракула, последняя критика Ревьюера, и передай решение пользователю. Ложный
успех (позеленело, но задача не решена) хуже честного «не смог».

## ФАЗА 4 — Завершение
1. Финальный прогон ВСЕХ гейтов проекта (не только целевого Оракула — не сломал ли соседнее):
   полный `npm run check`/`npm test`/`cargo test`/`build`, что применимо. Покажи вывод, не
   доверяй «зелёному» на слово.
2. `git diff --stat` и краткое резюме изменений.
3. Убери временную инфраструктуру (`task_spec.md`, любые сгенерированные скрипты-обёртки). **Тест
   Оракула, если ты его написал, по умолчанию ОСТАВЬ** — это регрессионный сторож; удаляй только
   явно одноразовые проверочные заглушки. Перед любым удалением — `git status`, чтобы не задеть
   чужое.
4. Реши по правилам репозитория: коммит/ветка/PR. Коммить, только если это авторизовано; не
   пушь/не мержи без отдельного разрешения.
5. Сообщи: «Автономный цикл завершён: Оракул зелёный, Ревьюер — APPROVE, все гейты пройдены»
   (или честный отчёт о не-сходимости) и остановись.

## Инварианты (не нарушать)
- Оракул проверяет поведение и заморожен от Кодера; переписывание тестов = провал, не успех.
- Один зелёный прогон — не доказательство; истина = детерминированный Оракул + независимый
  Ревьюер + финальный полный прогон гейтов.
- Контексты ролей физически разделены; обратная связь ограничена, история не копится.
- Каждая итерация чекпойнтится; не-сходимость откатывается и эскалируется человеку, не маскируется.
- Только разрешённые файлы; чужие незакоммиченные изменения неприкосновенны.

---

### Offline-fallback (ТОЛЬКО вне Claude Code, когда нативных субагентов нет)
Если исполняешь это без доступа к Agent/Workflow, тогда — и только тогда — сгенерируй два
самодостаточных скрипта на стандартной библиотеке (`urllib.request`), берущих ключ из переменной
окружения, без `pip install`. Корректные детали вызова (endpoint, заголовки `x-api-key` /
`anthropic-version`, id модели, формат Messages API) не выдумывай — возьми из навыка `claude-api`.
Всё равно соблюдай инварианты выше: заморозка Оракула, ограниченная обратная связь, чекпойнты,
эскалация. Удали скрипты в Фазе 4.
