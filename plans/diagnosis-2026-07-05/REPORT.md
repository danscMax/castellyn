## 🔴 СИСТЕМНЫЙ КОРЕНЬ №1 — залипшая синхронизация глушит ВСЁ приложение

**Это самый важный вывод. Один баг производит бОльшую часть жалоб «не работает / неактивно».**

Цепочка (доказана по коду):
1. Открытие вкладки «Синхронизация» авто-запускает `startSync('query')` (`+page.svelte:1299` `$effect`).
2. `startSync` ставит `running = 'sync'` (`:1293`) и зовёт `runSync(...).catch(...)` — fire-and-forget.
3. Rust `run_sync("query")` возвращает `Ok(0)` **мгновенно, минуя `spawn_streamed`** (`lib.rs:2431`) →
   событие `run-done` НЕ эмитится.
4. Но `running` снимается ТОЛЬКО в слушателе `run-done` (`+page.svelte:2181 if (running === id) running = null`).
5. → `running = 'sync'` **залипает навсегда**. Внизу вечно «● идёт: Синхронизация».
6. `busy = !!running` → каждый контрол с `disabled={busy}` по ВСЕМ вкладкам становится мёртвым.

**Что этим объясняется разом** (всё подтверждено — на скриншотах Профилей/Матрицы/MCP/Форков/Аналитики
внизу висит «идёт: Синхронизация»):
- мёртвое меню «три точки» в Профилях (`ProfilesTab.svelte:294..330` все `disabled: busy`);
- нередактируемая Матрица (`ProfilesMatrix.svelte:359,369,379,392,403` все `disabled={busy||applying}`);
- мёртвый MCP-таб (`McpTab.svelte:124..224` все `disabled={busy}`);
- неактивная «Синхронизировать wip-local» в Форках (частично; см. корень №5);
- частично неактивная «Применить» в Синхронизации;
- вечное «идёт: Синхронизация» в статус-баре везде.

**Root-cause фикс (маленький, высокая уверенность):** `run_sync` — обычная async-команда, а не
стримовая; момент разрешения её промиса И ЕСТЬ завершение. Снимать лок в `.finally`:
```js
function startSync(action, enabled?) {
  if (running) return;
  running = 'sync';
  log = [...];
  runSync(action, enabled).catch(onSpawnErr).finally(() => { if (running === 'sync') running = null; });
}
```
**Трейдофф:** нет. Стримовый `run_config_drift` (тоже `running='sync'`, но идёт через `spawn_streamed`
→ получает свой `run-done`) не затронут: `if (running) return` не даёт им пересечься, а `.finally`
снимает лок только если он всё ещё `'sync'`.

> ⚠️ Отдельно: гейт `busy = !!running` — сам по себе слишком грубый (любой прогон глушит ВСЁ, без
> объяснения «почему неактивно»). Даже после фикса залипания стоит: (а) не гасить не-конфликтующие
> действия на неродственном прогоне, ИЛИ (б) показывать тултип «идёт задача» вместо немого серого.
> Это корень №1b — см. раздел «Профили».

## 🔴 КОРЕНЬ №2 — ложное «2 с ошибкой» на Главной (доказано, живой Task Scheduler)

**Симптом:** Главная кричит красным «ЗАДАЧИ: 2 с ошибкой» / «проблем: 2», а внутри Расписания всё
зелёное «включено».

**Живая правда** (через `Get-ScheduledTask`): UpdateAll и Forks существуют, `State=Ready`,
`LastTaskResult = 267011 (0x41303)` = документированный сигнал Task Scheduler **«ещё ни разу не
запускалась»** (они назначены на завтра). Это НЕ ошибка.

**Корень (95%):** `Schedule-Hub.ps1:71` `$ok = ($lastResult -eq 0 -or $lastResult -eq 267009)` —
267011 (never-run) не входит → `ok=false`. Home ключуется на `ok` (`HomeTab.svelte:142
failing = tasks.filter(x => x.ok === false).length`) → «2 с ошибкой», красный. А ScheduleTab на `ok`
не смотрит вообще (`:56-62` бейдж только по `exists`/`enabled`) → зелёно. Отсюда расхождение.

**Root-cause фикс (в источнике, DRY):** `Schedule-Hub.ps1:71` → `$ok = ($status -notlike 'failed*')`
(строка `$status` уже декодирует never-run/ready/failed корректно). После этого never-run → `ok=true`,
`failing=0`, Home честен. ScheduleTab не затронут (он `ok` не читал).
**Трейдофф:** never-run перестаёт отличаться на Главной (это правильно — это не ошибка). Опционально —
нейтральный 4-й бакет «никогда не запускалась» как info, если хочется его видеть.

## 🔴 КОРЕНЬ №3 — «Бэкап конфигов: не создано» + клик «Создать» не срабатывает

**Живая проверка:** задача `ClaudeProfiles-Backup` действительно отсутствует в Task Scheduler (бейдж
честный). Логику создания я воспроизвёл вручную (`Schedule-Hub.ps1 -Action create -Id backup`) —
регистрируется штатно, exit 0. Значит стоячего бага «backup невозможно создать» НЕТ; проблема в том,
что провал создания у тебя стал **невидимым и неповторяемым**:

- **(A) 55% — ложный успех:** `Schedule-Hub.ps1:119` ловит исключение `Register-ScheduledTask`, пишет
  красную строку и **всё равно** доходит до `Write-SchedulesJson` + exit 0. Rust отдаёт exit 0, UI
  тостит «готово», а `reloadSchedules` тут же возвращает карточку в «не создано» → ровно «нажал
  создать, осталось не создано, без ошибки». **Фикс:** в `catch` — `exit 1` после лога (+ проверить,
  что задача реально существует после Register). Трейдофф: enable/disable/run/delete делят этот catch —
  тоже станут честно репортить провал (это хорошо).
- **(B) 30% — клик проглочен, т.к. лок занят:** `+page.svelte:1605 if (running) return;` — без тоста.
  Если что-то держит `running` (в т.ч. залипшая синхронизация из корня №1!), клик «Создать» — немой
  no-op. **Фикс:** тост «идёт задача» как на `:1976`.
- (C) 15% — исторически backup промахнулся при первичной пакетной регистрации; чинится (A)/(B).

## 🟠 КОРЕНЬ №4 — «Применить» в Синхронизации неактивна, хотя плашка велит её нажать

**Симптом:** жёлтая плашка «требует применения — …нажмите Применить», но кнопка «Применить» (внизу, под
тумблерами) серая и не подсвечена.

**Корни:**
- Частично — корень №1 (залипший `busy`).
- Независимо: `SyncTab.svelte:278 disabled={busy || !dirty}`, где `dirty` = только локальные правки
  тумблеров. А плашка загорается на `stignoreMatches===false` (развёрнутый .stignore разошёлся с
  сохранённым) — это ДРУГОЕ условие. Т.е. когда нужно применить именно из-за дрейфа .stignore, `dirty`
  ложно → кнопка серая.

**Root-cause фикс:** `disabled={busy || (!dirty && data.stignoreMatches !== false)}`. И UX: положить
копию кнопки «Применить» прямо в плашку (переиспользовать `common.apply`, без новых i18n) — плашка
должна нести действие, а не отсылать к серой кнопке внизу. Трейдофф: нет.

## 🟠 КОРЕНЬ №5 — Форки: «not-a-fork», неактивный wip-sync, потраченное место

- **not-a-fork (bifrost, ccs) — детекция КОРРЕКТНА (95%)** (`ForkSync.psm1:483`): это git-репы под
  root-ом `E:\Scripts\External`, у чьего GitHub-origin нет parent и которые не в `ownPaths`. Реальный
  дефект — catch-all скан папки плодит мёртвые неактивные карточки. **Фикс:** свернуть их в группу
  «Не форки (N)» (как существующую GitHub-only секцию) + дать возможность reclassify/exclude (см.
  де-хардкод ниже). Код детекции менять не надо.
- **«Синхронизировать wip-local» серая, хотя рекомендована (90%):** кнопка гейтится на ГЛОБАЛЬНЫЙ слот
  (`ForkRepoCard.svelte:178 rec.disabled = anyRunning || busy`), а per-repo действия используют
  НЕЗАВИСИМЫЙ слот (`run_fork_repo`, `lib.rs:1101/1125`), который бэкенд не гейтит на неродственные
  прогоны. Плюс залипшая синхронизация (корень №1) держала `anyRunning`. **Фикс:** гейтить на
  `busy || running === 'forks'` (то, что реально блокирует бэкенд), а не `anyRunning`. Добавить
  недостающий `!repo.isOwn`-guard в ветку рекомендации syncwip/delwip. Трейдофф: нет (бэкенд и так
  обеспечивает взаимоисключение).
- **Layout (95%):** `app.css:423 grid-template-columns: repeat(auto-fill, minmax(330px, 1fr))` → мало
  колонок + растяжка 1fr = «широкие и пустые»; имя в `ForkRepoCard.svelte:253` конкурирует с чипами
  `фрок`/`own` и `health` на одной строке → «FreeQw…». `align-items:start` + grid-строки = masonry-щели.
  **Фикс:** `minmax(270px,1fr)` (4-5 колонок), имя на свою строку (`break-words` + `title`, чип в
  подзаголовок), опц. компакт-режим/DataTable для многих репо. Трейдофф: узкие карточки — ветки уже
  `flex-wrap`, безопасно.

## 🟡 КОРЕНЬ №6 — Аналитика только по freellmapi; статуслайн в заголовок

- **Аналитика (100%, дизайн-пробел):** `read_freellmapi_analytics` читает единственный источник
  `freeapi.db` (`lib.rs:2878`). Данные для «всей аналитики» (сессии, время, токены по профилям) лежат в
  других местах, которые вкладка не агрегирует (OAuth-лимиты `limits.rs`, `history.jsonl`/runHistory,
  `agent_status.rs`, история расписания). Плюс гистограмма «История запусков» вечно пустая, т.к.
  `runHistory` — in-memory стор, не персистится. **Фикс:** мульти-источниковый дашборд с переключателем
  (Шлюз / Claude-usage / Обслуживание / Сессии); персистить runHistory. 20 метрик — в
  `diag-analytics-sessions.md`. Трейдофф: точные «токены по Claude-профилю» недостижимы (OAuth даёт %
  утилизации, не сырые токены) — только приближение.
- **Статуслайн → заголовок (дизайн):** in-pane статуслайн рисует сам Claude Code внутри PTY — Castellyn
  его «перенести» не может, только продублировать НАТИВНОЙ полосой статуса для сфокусированной сессии
  (из `agent_status`+`limits`+профиль/tool/cwd) в кастомном заголовке, с настройкой вкл/выкл. Скрейпить
  терминал — хрупко, не рекомендуется.

## 🧭 СКВОЗНАЯ ТЕМА — ДЕ-ХАРДКОД для нового пользователя «с нуля»

Владелец прав: множество вещей заточено под ЕГО машину. Что нашли:

| Область | Что захардкожено | Root-cause направление |
|---|---|---|
| Расписание | Фиксированная таблица 5 задач `$defs` (имена/скрипты/аргументы/времена) `Schedule-Hub.ps1:30`; путь форков расходится с манифестом | Читать из декларативного реестра (манифест + user `schedules.json` в `%APPDATA%\castellyn`); имя задачи по умолчанию `Castellyn-<id>` |
| Форки | Discovery уже конфиг-driven (`repos.json`: roots/paths/ownPaths), но БЕЗ UI и в vendored `tools/` (клобберится при обновлении) | Вынести конфиг в `%APPDATA%\castellyn` + IPC read/write + UI (add/remove/set-root) + «Сканировать папку…» → чеклист. Whole-PC скан — только по явной кнопке |
| MCP | `ALL_PROFILES` фолбэк `['ccmy','cc1'..'cc5']` + `PLUGIN_PROVIDED=['context7','serena']` `McpTab.svelte:77,81` | Прокинуть реальный список профилей пропом; не рисовать канистру имён |
| Синхронизация | Путь `!Настройки и MCP\ClaudeProfiles\config\…` под (конфигурируемым) scripts_root | Конфиг `profilesConfigDir` (follow-up) |
| Профили | Зеро-стейт УЖЕ работает (OnboardingWizard + «+ Добавить профиль» + EmptyState); имена НЕ хардкожены (кроме MCP-фолбэка выше) | Только починить MCP-фолбэк |

**Итог по де-хардкоду:** инфраструктура частично есть (манифест, repos.json, onboarding), но не выведена
в UI и местами лежит в клоббер-опасных vendored-файлах. Направление: пользовательский конфиг → durable
`%APPDATA%\castellyn`, реестры читаются с диска, UI для add/remove/scan, дефолты самоименуются.

## Приоритизация фиксов (Impact × уверенность × Effort)

**🔥 Волна A — системные баги, малый диф, высокая уверенность (чинить первыми):**
1. Корень №1: `.finally` снимает `running` в `startSync` — оживляет меню/матрицу/MCP/форк-кнопки разом.
2. Корень №2: `Schedule-Hub.ps1` `ok` = не-`failed*` — убирает ложное «2 с ошибкой».
3. Корень №3A: `exit 1` в catch создания задачи — конец ложного успеха.
4. Корень №4: `disabled` учитывает `stignoreMatches` + кнопка «Применить» в плашке.
5. Корень №5: форк-кнопки гейтить на `running==='forks'`, не `anyRunning`; not-a-fork в свёрнутую группу.
6. Таблица Профилей: сузить fixed-колонки/адаптив (не лезет на экран).
7. MCP `cc*`-фолбэк → реальные профили.

**⭐ Волна B — легибельность busy-гейта:** не гасить не-конфликтующие действия / тултип «идёт задача».

**📋 Волна C — де-хардкод и фичи (крупнее, отдельный заход):** UI-конфиг форков + скан папки; реестр
расписания + user schedules.json; мульти-источниковая Аналитика + персист runHistory; нативная полоса
статуса сессии в заголовке; редизайн Матрицы (громкий dirty + плавающий Apply).
